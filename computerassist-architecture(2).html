Component Reference

Detailed explanation of every component in the ComputerAssist architecture.

Layer 1 — Input
#01

OS Events Collector

DoesMonitors all OS-level activity: filesystem changes (inotify/fanotify), keyboard and mouse input (evdev), process spawns, app launches, and window focus changes. Broadest, lowest-level view of user activity.
InputRaw OS event streams from kernel APIs (inotify, fanotify, evdev, proc).
OutputStructured OS event records → Feature Extraction Layer.
DetailLinux-native APIs for zero-overhead, purely event-driven capture. No polling — inotify for file events, fanotify for filesystem-wide monitoring, evdev for kernel-level input devices.
#02

Shell / Terminal Collector

DoesHooks into zsh via precmd/preexec to capture every shell command before and after execution. Records full command string, exit code, duration, and working directory.
InputZsh hook callbacks (precmd/preexec), shell environment variables.
OutputCommand event records (command, args, exit code, duration, cwd) → Feature Extraction.
Detailprecmd fires before each prompt (post-execution data), preexec fires before command execution (pre-execution data). Together they bracket every command with full before/after context.
#03

WezTerm Collector

DoesHooks into WezTerm's Lua API to capture pane splits, tab creation, window events, and cross-environment session state (WSL + PowerShell panes simultaneously).
InputWezTerm Lua event API callbacks.
OutputTerminal session topology events → Feature Extraction.
DetailWezTerm's Lua API exposes session structure that tmux/screen cannot — particularly the ability to see both WSL and native Windows environments in a single event stream.
#04

Context Signals Collector

DoesCaptures ambient context: battery %, WiFi SSID, GPS/location, time of day, day of week. Enables pattern disambiguation — same action may mean different things at home vs. work.
InputSystem APIs for battery, network, location, system clock.
OutputContext signal records → Feature Extraction.
DetailDisambiguation features — they don't trigger predictions themselves but allow the prediction layer to differentiate superficially identical sequences in different environments.
#05

App Integrations

DoesCommunity-extensible plugin API. Individual apps (browser, editor, etc.) emit structured events through registered plugins, enabling in-app action understanding beyond OS-level monitoring.
InputPlugin callbacks using the standard event schema.
OutputApp-specific structured events → Feature Extraction.
DetailStandard schema any app can implement. Extensible without core modifications — community adds app support, events flow through the same pipeline as all other sources.
#06

★ Feature Extraction Layer

DoesMost critical component. Transforms all raw events into structured, weighted feature vectors. Separates base commands from arguments (z ~/path → {cmd:"z", weight:HIGH} + {arg:"~/path", weight:LOW}). Assigns type-based weights, normalises to unified schema, tags with session context.
InputRaw event streams from all five collectors (heterogeneous formats).
OutputUnified Event Stream — structured, weighted feature vectors → HMM, VMM, and Log Shipping.
DetailEvery downstream model depends on this layer. Weight assignment here (HIGH for structural commands, LOW for variable arguments) enables fuzzy matching later. Single point where signal quality is determined for the entire system.
Layer 2 — Prediction
#07

HMM — Hidden Markov Model

DoesDiscovers hidden workflow states entirely unsupervised — states like "deployment session" or "coding startup" emerge from data without labels. Assigns current stream to a workflow state with confidence %.
InputUnified Event Stream from Feature Extraction.
OutputCurrent workflow state + confidence % → VMM.
DetailTrained overnight on Personal Server via Baum-Welch (EM for HMMs) — too compute-heavy for the laptop. Has its own per-feature dynamic weight system. Local copy updated daily via Model Sync.
#08

VMM — Variable-Order Markov Model

DoesCore prediction engine. Receives event stream + HMM state, predicts next action. Adaptive lookback depth (optimal history length per context, not fixed N). Continuous online learning — updates on every event.
InputUnified Event Stream + HMM's (workflow state × confidence).
OutputNext-action prediction + probability → Fuzzy Pattern Store (prediction) and Confidence Aggregator (probability).
DetailHMM influence: effective_weight = VMM_trust × HMM_confidence. Uncertain HMM auto-dampens. Per-pattern, per-feature dynamic weights. Immediate online updates from Feedback Collection.
#09

Fuzzy Pattern Store

DoesCustom confidence validation. Stores confirmed recurring sequences with per-sequence weight profiles. Fuzzy weighted matching: high-weight features must match exactly, low-weight features are flexible.
InputVMM's prediction.
OutputPattern boost signal → Confidence Aggregator. Strong boost if mid-sequence.
Detailz ~/projectA and z ~/projectB match the same pattern: z (HIGH) matches exactly, path (LOW) is flexible. Mid-sequence bonus is critical for completion confidence.
#10

Confidence Aggregator

DoesCombines VMM probability + Fuzzy Pattern Store boost (including mid-sequence bonus) into single final confidence score. Applies threshold — only above-threshold predictions pass forward.
InputVMM probability + pattern boost signal.
OutputFinal confidence score → Phase Gate (only if threshold met).
DetailSafety gate. Even high-probability VMM predictions only proceed if the combined score after pattern validation exceeds the threshold.
#11

Phase Gate

DoesManual user-controlled mode switch. Routes predictions to Training Mode or Executing Mode. User flips when they've observed enough accuracy in the Taskbar Widget.
InputAbove-threshold predictions + user's mode selection.
OutputPrediction routed to Training Mode or Executing Mode.
DetailDeliberately manual — system never auto-promotes to execution. Trust mechanism: observe in training, build confidence, then enable.
Personal Server (Sidebar)
#12

Log Shipping

DoesPeriodically ships structured event logs from laptop to personal server over local network. Also receives feedback/correction data from Feedback Collection as additional training input.
InputEvent stream logs from Feature Extraction + feedback data from Feedback Collection.
OutputAccumulated structured logs → HMM Batch Training.
DetailStrictly local network — no cloud. Logs include raw events and feedback signals so HMM learns from both observed patterns and user-validated outcomes.
#13

HMM Batch Training

DoesRuns Baum-Welch on accumulated logs overnight while laptop is off. Discovers and refines hidden workflow states from full behaviour history.
InputAccumulated logs from Log Shipping.
OutputTrained HMM model parameters → Model Sync.
DetailBaum-Welch is computationally intensive (EM over full dataset). Overnight on server means laptop bears no cost. Each run incorporates feedback corrections.
#14

Model Sync

DoesOn laptop's first startup each day, pulls latest trained HMM from server and replaces local copy.
InputTrained HMM model from HMM Batch Training.
OutputUpdated HMM model → HMM (dashed feedback arrow).
DetailOnce per day on first boot. HMM is stable throughout each day, gets improved version next morning. Daily cadence matches overnight training cycle.
Layer 3 — Execution
#15

Training Mode

DoesSafe observation phase. Predictions computed and displayed but never executed. Full pipeline runs silently, results shown in Taskbar Widget.
InputAbove-threshold predictions from Phase Gate (training mode).
OutputLive prediction + confidence → Taskbar Widget.
DetailWhere user builds trust — they work normally while watching. Accuracy observation leads to confidence for the switch to Executing Mode.
#16

Taskbar / Topbar Widget

DoesDisplays live prediction and confidence in system taskbar. Persistent, non-intrusive window into system state.
InputCurrent prediction + confidence from Training Mode.
OutputObservation signals → Feedback Collection (shadow mode data).
DetailAlways visible but unobtrusive. Primary interface for judging model readiness. Active in both modes; in Training Mode it's the only output.
#17

Executing Mode

DoesActive automation phase. Routes predictions through Suggestion UI then Executor for OS-level execution.
InputAbove-threshold predictions from Phase Gate (executing mode).
OutputPredictions → Suggestion UI → Executor.
DetailUser-controlled. Even here, Suggestion UI provides cancel window — never fully unsupervised.
#18

Suggestion UI

DoesShows readable description of pending action. Cancel shortcut + 2–3 second countdown. Mid-sequence: instant completion (skips countdown, user actions confirm intent).
InputPredicted action sequence from Executing Mode.
OutputConfirmed action → Executor.
DetailCountdown is last line of defence. Mid-sequence instant completion: if steps 1–3 of 5 done, system completes 4–5 immediately.
#19

Executor

DoesCarries out action sequences at OS level: launch apps, run commands, file ops, system actions. Injects commands into WezTerm for terminal workflows. Future per-app actions via plugin API.
InputConfirmed action from Suggestion UI.
OutputExecution results → Feedback Collection.
DetailOS-level for max capability. WezTerm injection uses same Lua API as Collector — symmetric read/write loop.
#20

Feedback Collection

DoesReceives signals from Widget + Executor. Four modes: shadow (continuous silent comparison), accept (strong positive), reject pre-exec (negative + logs show correction), undo post-exec (negative + logs show correction).
InputSignals from Taskbar Widget and Executor.
OutputThree loops: (1) VMM online update, (2) FPS pattern validation, (3) Log Shipping for HMM retraining.
DetailEvent log always captures ground truth. Shadow mode runs in both modes — system always self-evaluates. Reject/Undo are most valuable: subsequent user actions reveal what should have happened.
#20b

Feedback Loops (Dashed Arrows)

OverviewThree dashed feedback paths carry learning signals from Feedback Collection back upstream, closing the loop so the system continuously self-improves from every interaction.
→ VMMImmediate online weight update. Every feedback signal (accept, reject, undo, shadow comparison) triggers a real-time adjustment to the VMM's per-pattern and per-feature weights. This is how the VMM learns continuously without retraining — it adapts on every single event.
→ FPSPattern validation update. Confirms or invalidates stored sequences in the Fuzzy Pattern Store. Accepted predictions strengthen a pattern's confidence; rejected or undone predictions weaken or remove it. This keeps the pattern library accurate over time.
→ ServerFeedback and correction data shipped to Log Shipping as additional training input for the next HMM batch training run. This means the HMM doesn't just learn from raw event logs — it also learns from the user's explicit accept/reject/undo signals and the corrected action sequences that follow.
Core System — Adaptive Weight Engine
#21

Weight System (Adaptive, Per-Feature, Per-Pattern)

OverviewCross-cutting adaptive weight system inside HMM, VMM, and Fuzzy Pattern Store. Each component maintains per-feature and per-pattern dynamic weights evolving from accuracy and feedback.
BaseInitial weights by type in Feature Extraction: structural features (commands, app names) → HIGH; variable features (paths, args) → LOW. Establishes the signal-to-noise hierarchy.
In HMMPer-feature weights in emission/transition matrices. Strong workflow-state indicators amplified; noisy features dampened. Updated during overnight Baum-Welch training.
In VMMPer-pattern and per-feature weights. Each pattern has its own profile; features weighted by predictive value. Updated immediately (online) — accepts boost, rejects/undos decrease.
In FPSPer-sequence weight profiles control fuzzy matching. z ~/projectA and z ~/projectB match because command (HIGH) matches exactly, path (LOW) is flexible. Updated from Feedback Collection.
FeedbackAll updates flow through Feedback Collection. Accepts strengthen weights. Rejects/Undos reduce influence of error-causing features. Self-correcting: bad predictions shift weight profiles away from the error.
